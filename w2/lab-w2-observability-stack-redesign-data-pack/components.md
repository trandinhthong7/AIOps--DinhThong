# Component Decision Table

Target principle: keep one on-call query surface, reduce log/index spend first, preserve trace depth for RCA, and keep paging reliable while moving correlation upstream.

| Capability | Chosen component/vendor | Why this one | What gets worse if we change our mind in six months |
|---|---|---|---|
| Metrics ingestion + storage + query | OpenTelemetry Collector -> VictoriaMetrics Cluster + vmalert | VictoriaMetrics handles Prometheus-style metrics with lower operational complexity than a full Mimir deployment for a 10-service/300-host estate, and it gives explicit control over high-cardinality labels. | Migrating to Mimir/Datadog later means rewriting recording rules, dashboards, and alert queries after teams have learned MetricsQL/PromQL patterns. |
| Logs ingestion + storage + search | OTel Collector/Filelog -> Grafana Loki hot tier + S3 compressed archive | Logs are the biggest low-value cost lever; Loki keeps service/severity/incident labels searchable while moving raw long-tail retention to cheap object storage. | If search needs Splunk-style full text analytics again, older archived logs will be slower to hydrate and analysts lose some SPL workflows. |
| Distributed tracing | OTel traces -> Grafana Tempo with tail sampling | The current 1% sampling hides rare slow requests; Tempo with OTel tail sampling can retain 100% of error/slow traces and 10% baseline without paying APM-host tax. | If Tempo is replaced later, sampling policies and trace-ID links in dashboards/runbooks must be ported carefully or RCA quality regresses. |
| Alerting + correlation/grouping | vmalert + in-house correlation service using service graph/fingerprints | Per-service alerting already works; missing capability is grouping 47 pages into one incident and attaching graph context. | Moving to a SaaS AIOps correlator later may duplicate grouping semantics and make the incident evidence ledger harder to keep authoritative. |
| Incident routing + paging | PagerDuty Business retained, seat count reduced from 65 to 35 | Paging is not the main cost driver and is operationally trusted; keep it while sending fewer, grouped incidents. | Replacing PagerDuty during the observability migration creates training and notification risk with little diagnostic-speed upside. |
| Dashboards + SLO tracking | Grafana OSS/Enterprise-light deployment backed by VM/Loki/Tempo | Grafana becomes the single human query surface for metrics, logs, traces, SLO burn rates, and incident evidence links. | If we move away from Grafana later, every dashboard and Explore link embedded in runbooks/postmortems must be rebuilt. |
| Synthetic checks | Blackbox exporter + k6 smoke checks from two regions | Current 270 Datadog synthetic checks are broader than needed; keep customer-critical checks and run them through the same metrics/alerting path. | Datadog Synthetics has richer managed geography and browser workflows; replacing it later means re-authoring checks. |
| Incident decision audit | In-house evidence ledger backed by Postgres/S3 JSONL export | Pain points 7 and 9 need queryable decisions/actions, which vendors do not own consistently across all signals. | If this is cut, postmortems remain Slack archaeology and the MTTR-learning loop stays manual. |
