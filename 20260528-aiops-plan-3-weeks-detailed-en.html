Self-study format: Each self-study day has a technical guide (~700 lines) with exercises embedded directly in the content. Students read → do exercise → read more → do next exercise. Week 1 — Detection & Triage

"Save the burning system" — detect anomalies + parse logs from real telemetry
Tue AM Incident Walkthrough — Week Kickoff LIVE 2–2.5h
Content
  • Walk through a real incident (e.g. Cloudflare 2022, AWS us-east-1, GitHub outage) — trigger → triage → RCA → post-mortem
  • AIOps problem taxonomy + mapping problems ↔ techniques
  • Week briefing + distribute materials + Q&A
D1 (Mon) Metric Anomaly Detection SELF-STUDY
Topics
  • Foundations: normal distribution, skewness, stationarity — why they matter for anomaly detection
  • Statistical: Rolling Z-score (3σ), EWMA, STL decomposition — mechanisms, when to use, when they fail
  • ML: Isolation Forest (intuition, feature engineering, tuning), One-Class SVM (comparison)
  • DL: Autoencoder, LSTM-AE — intuition, when needed, trade-offs vs simpler methods
  • Univariate vs Multivariate — why multivariate catches memory leaks that univariate misses
  • Feature engineering for time series: rolling stats, lag, rate of change, temporal features
  • KPIs: precision, recall, F1, TTD, false alarm rate — recall > precision in AIOps
Embedded exercises
  • EDA: load NAB data, compute skewness, plot ACF, determine data type
  • Implement rolling Z-score + observe results on specific data
  • Implement Isolation Forest with feature engineering + tune contamination
  • Compare 2 detectors: precision/recall table + explain trade-offs
End-of-day checkpoint: Notebook with all exercises completed + SUBMIT.md reflection (data type, method choice, reasoning, trade-offs).
D2 (Tue PM + Wed) Log Mining + Parsing + Log-based Anomaly Detection SELF-STUDY
Topics
  • Why logs matter: metrics tell "what's wrong", logs tell "why"
  • Log parsing problem: millions of unique log lines → need template extraction
  • Drain3 deep dive: parse tree algorithm, fixed-depth tree, template extraction
  • Parser comparison: Drain3 vs Spell vs Lenma vs LLM-based — when to use which
  • Log embedding: TF-IDF on templates, sentence-transformers for semantic similarity
  • Anomaly detection on logs: template count spikes, new template detection, semantic drift
  • Combining metrics + logs: when logs are sufficient, when cross-signal is needed
  • Production considerations: log volume, sampling, storage cost, structured vs unstructured
Embedded exercises
  • Parse HDFS logs with Drain3, observe template tree, explain output
  • Create template count time series → apply anomaly detector from D1
  • TF-IDF on templates → find similar/dissimilar clusters
  • New template detection: inject unknown log line → detect
Checkpoint: Notebook exercises + top-10 templates CSV + SUBMIT.md (parser comparison, log vs metric insights).
D3 (Wed) Data Layer Architecture + Observability Pipeline SELF-STUDY
Topics
  • Observability pipeline end-to-end: collection → transport → processing → storage → query
  • OpenTelemetry (OTel): traces, metrics, logs — unified collection, SDK + Collector architecture
  • Transport layer: Kafka vs direct push — when you need a message queue, when you don't
  • Feature store: online vs offline — when needed for AIOps, when overkill
  • Streaming feature pipeline: Kafka + Flink/Spark Streaming for real-time feature extraction
  • Schema registry + data contracts: preventing "log format changed → pipeline breaks"
  • Storage trade-offs: time-series DB (InfluxDB, Prometheus) vs data lake (S3 + Parquet) vs both
  • Cost model: storage, compute, egress — capacity planning for observability data
Embedded exercises
  • Draw architecture diagram for an AIOps use case (Python diagrams library)
  • Mock streaming pipeline: Python consumer reading from queue → extract features → store
  • Cost estimation: calculate storage cost for 1 month of metric + log data
  • ADR: write an Architecture Decision Record for "Kafka vs direct push"
Checkpoint: Architecture diagram PNG + pipeline script + cost estimation + ADR + SUBMIT.md.
Thu Individual Lab — Detect & Triage on "Broken System" OFFLINE
Opening (30 min)
Spot-check random 3 students (90s each) + Q&A blockers.
Scenario
"Broken system" dataset — a simulated failing microservice with 24h of metrics + logs. Find: when the failure started, where it originated, what the symptoms were.

Input

  • Dataset: metrics + logs (~500MB)
  • Repo template aiops-w1-{name}
  • 1-page scenario brief

Output

  • Anomaly detection pipeline on metrics
  • Log clustering with Drain3
  • Narrow down: timestamp + service + root cause hypothesis

Deliverable package (end of Thu)

  • notebooks/01-eda.ipynb
  • notebooks/02-anomaly-detection.ipynb
  • notebooks/03-log-clustering.ipynb
  • FINDINGS.md — conclusion: when, where, root cause hypothesis
Fri Wrap — Pod Show-and-Tell + Random Viva CHECKPOINT
  • Morning: finalize deliverables
  • 13:30–15:00: show-and-tell in pods of 5 (5 min present + 3 min Q&A), pod feedback sheet
  • 15:00–15:45: random viva — 5–7 students present to entire class
  • 15:45–16:30: buffer + escalation (mentor 1:1 with struggling students)
Submit by end of Fri: Deliverable package + pod feedback sheet.

Week 2 — RCA & Smart Response

"Understand why" — build a mini RCA assistant + correlation pipeline
Tue AM Alert Storm Walkthrough — Week Kickoff LIVE 2–2.5h
Content
  • Walk through an alert storm: 1 incident generating 200 alerts in 5 minutes
  • Live demo: alert correlation compressing 200 → 12 clusters, RCA assistant answering "what broke + why"
  • Pattern: Detect → Enrich → Decide → Act → Verify
  • Week briefing + distribute materials + Q&A
D1 (Mon) Alert Correlation + Noise Reduction SELF-STUDY
Topics
  • Alert fatigue: why 200 alerts/5 min = useless, on-call ignores them
  • Dedup, suppression, grouping — rule-based correlation
  • Topology-aware correlation: service dependency graph → group alerts by service chain
  • Temporal correlation: sliding window, co-occurrence matrix
  • ML-based alert clustering: feature engineering, DBSCAN, hierarchical clustering
  • Noise reduction metrics: compression ratio, precision, coverage
  • Causal vs correlational — common trap: "alert A before B" ≠ "A caused B"
  • Production: PagerDuty/Opsgenie integration, alert routing, escalation policies
Embedded exercises
  • Rule-based correlation: implement dedup + suppression on alert dataset
  • Temporal clustering: sliding window → co-occurrence → group alerts
  • Topology-aware: load service graph → cluster alerts by dependency chain
  • Evaluate: 200 alerts → how many clusters, compression ratio
Checkpoint: Notebook exercises + cluster_summary.json (200→≤20 clusters) + SUBMIT.md.
D2 (Tue PM + Wed) RCA — Graph, Causal + LLM-augmented SELF-STUDY
Topics
  • RCA problem: from "what's wrong" → "why it's wrong" — why it's hard
  • Service dependency graph: build from trace data, topology discovery
  • Graph-based RCA: propagation analysis, MonitorRank, random walk on service graph
  • Hierarchical metric explanation: top-down drill from high-level KPI → root metric
  • Causal inference basics: PC algorithm, Granger causality — intuition only
  • LLM-augmented RCA: incident summarization, runbook RAG, prompt patterns for RCA assistant
  • Hallucination risk: LLMs can fabricate root causes — validation and guardrails
  • Case studies: Netflix RCA, MonitorRank paper, Datadog RCA
Embedded exercises
  • Build service graph from JSON → visualize with networkx
  • Walk graph: given anomaly alert, trace upstream/downstream → find root service
  • Mini RCA assistant: RAG over service graph + runbook → answer "what + why"
  • Evaluate: groundedness check — does the RCA answer cite actual data?
Checkpoint: Notebook exercises + FINDINGS.md (RCA for 3 scenarios) + SUBMIT.md.
D3 (Wed) Model Serving & ML Deployment Architecture SELF-STUDY
Topics
  • Inference patterns: batch vs online vs streaming — latency budget per AIOps use case
  • Serving stack deep dive: BentoML, Triton, KServe, Seldon — architecture comparison
  • Model registry: MLflow — versioning, staging, rollback, lineage tracking
  • Deployment patterns: canary, shadow, A/B — when to use which for ML models
  • Monitoring ML in production: data drift, concept drift, performance degradation
  • GPU inference optimization: batching, quantization, model distillation — overview
  • Cost model: CPU vs GPU inference, cold start, scaling strategy
  • Architecture pattern: model serving for anomaly detection pipeline (real-time + batch)
Embedded exercises
  • Wrap anomaly detector (from W1) as FastAPI /predict endpoint
  • Test endpoint: curl request → verify response format
  • Architecture diagram: model serving pipeline for an AIOps use case
  • Cost estimation: CPU vs GPU inference for 1M predictions/day
Checkpoint: serve.py + DESIGN.md (serving architecture) + cost estimation + SUBMIT.md.

  • FINDINGS.md — RCA for 3 scenarios
Fri Wrap — Pod Show-and-Tell + Random Viva CHECKPOINT
Same format as Week 1: morning finalize, afternoon pod show-and-tell + random viva + buffer.
Submit by end of Fri: Deliverable package + pod feedback sheet.

Week 3 — Auto-remediation, Closed-loop & E2E Platform

"Put AI into production" — enough architecture to design end-to-end and write specs clear enough to collaborate with ops teams
Tue AM E2E Platform Walkthrough — Week Kickoff LIVE 2–2.5h
Content
  • Walk through a real platform (Datadog Watchdog / Dynatrace Davis / Splunk AIOps) — layer by layer
  • Live demo: closed-loop scenario — high latency → AI suggests scale → human approves → execute → verify
  • Why closed-loop is hard: idempotency, blast radius, human-in-the-loop gates
  • Week briefing + distribute materials + Q&A
D1 (Mon) Auto-remediation & Safety Guardrails SELF-STUDY
Topics
  • Remediation maturity: manual → rule-based → ML-recommended → autonomous
  • Action catalog: scale, restart, failover, throttle, redirect traffic, circuit break
  • Decision engine design: rule engine + ML-recommended hybrid
  • Safety guardrails deep dive: blast radius estimation, dry-run, circuit breaker, rollback policy
  • Decision framework: when autonomous is OK, when human-in-the-loop is required
  • GenAI runbook: LLM generates step-by-step remediation → human approves → execute with guardrails
  • Audit trail: every action must be logged, traceable, reviewable
  • Case studies: PagerDuty auto-remediation, Datadog Watchdog remediation
Embedded exercises
  • Build decision engine: ≥3 rules + ≥1 ML-recommended action
  • Implement dry-run mode: execute actions but only log, don't actually change
  • Audit log: JSON log of every decision (input → rule matched → action → outcome)
  • Safety checklist: evaluate an action catalog using blast radius framework
Checkpoint: decide.py + audit_log.json + safety checklist + SUBMIT.md.
D2 (Tue PM + Wed) Closed-loop Architecture + Workflow Orchestration SELF-STUDY
Topics
  • Closed-loop pattern: Detect → Enrich → Decide → Act → Verify — why a loop, not a pipeline
  • Event-driven backbone: Kafka vs NATS vs Pulsar — when to use which
  • Workflow orchestration: Argo Workflows / Temporal / Step Functions — comparison
  • Idempotency + exactly-once for remediation actions — why it's critical
  • State machine design: states, transitions, human-in-the-loop gates, timeout handling
  • Verify step: post-action outcome checking — success criteria, rollback triggers
  • Feedback loop: verify results improve detect + decide for next time
  • Reference: Netflix auto-diagnosis, AWS re:Invent closed-loop talks
Embedded exercises
  • Design state machine for a remediation scenario (diagram)
  • Build pipeline: detect→enrich→decide→execute(dry)→verify, run E2E on 1 scenario
  • Implement verify step: check outcome metric after action
  • Failure scenario: action fails → rollback → log → alert
Checkpoint: pipeline.py (5-stage) + state machine diagram + SUBMIT.md.
D3 (Wed) E2E AIOps Platform Design + Cross-team Spec SELF-STUDY
Topics
  • E2E AIOps platform: OTel → Kafka → Flink → Feature store → Model serving → Decision engine → Executor
  • Build vs Buy framework: when to build, when to buy (Datadog, Splunk, PagerDuty, custom)
  • Cost model: storage, compute, GPU inference, data egress — monthly estimate per scale
  • Writing specs for ops teams: SLO/SLI, alert budget, infra requirements, deployment expectations
  • ADR template: context, decision, consequences, alternatives considered
  • Trade-off framework: cost ↔ latency ↔ accuracy ↔ complexity — quantify trade-offs
  • Team boundaries: AI team vs Platform team vs Ops team — interface contracts
  • Reference architecture: Datadog, Splunk, open-source stack
Embedded exercises
  • E2E architecture diagram (Python diagrams library) for one use case
  • Spec document: infra + deploy + SLO requirements for ops team
  • ADR: ≥1 major architecture decision with alternatives + trade-offs
  • Cost model script: estimate monthly cost for 3 tiers (small/medium/large)
Checkpoint: Architecture diagram + SPEC.md + ADR.md + cost_model.py + SUBMIT.md.
